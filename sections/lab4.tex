\section{Lab exercise 4: 2D-Filter and Image Processing}

Image $P(x, y)$ is 2 dimensional signal with spatial coordinates $x$ and $y$. Signal values in this case are intensity (brightness) values of pixels ranging from 0 to 255 captured by sensor. Some captured by digital sensor rays are very strong and cause maximum intensity 255, equivalently to white color. Some areas on the sensor do not capture any rays and thus can get intensity value of 0, equivalently complete black color. 

2 dimensional signals can be modified and filtered in time and frequency domains following same principles as 1 dimensional signals by applying DFT in both spatial directions $x$ and $y$ sequentially, first DFT in $x$ direction (column by column), and second applied to the result of first in $y$ direction (row by row). The result is $Q(u, v)$ frequency spectrum that has same size as image with DC component at $Q(0, 0)$ representing average intensity of image, and values corresponding to bigger $u$ or $v$ represent high frequency details in image like tiny edges. Of course, conjugate symmetry applies but with 4 quadrants in frequency spectrum, as all properties for 1 dimensional DFT hold. Each $(u, v)$ pair is information about containment of sinusoid with that specific frequency in image (Figure~\ref{fig:401}).

\begin{figure}[H]
  \includegraphics[width=0.6\textwidth]{image_103.png}
  \caption{frequency spectrum}\label{fig:401}
\end{figure}

Labs 4 and 5 do not focus on frequency spectrum of images, instead use 2 dimensional kernels, thus I will not go into much more detail. What is important to mention is that frequency in image is basically amplitude change between pixel intensities of neighboring pixels (Figure~\ref{fig:402}). If 1 pixel has value 0 = Black, and second pixel receives ray with value 255 = White, then this is highest frequency possible. Amplitude change is from lowest to highest value (white edge on black background). While, smooth changes like sky gradient would correspond to low frequency components. 

\begin{figure}[H]
  \includegraphics[width=0.48\textwidth]{image_102.png}
  \caption{high frequency amplitude change}\label{fig:402}
\end{figure}

Same logic as with 2 dimensional DFT applies also to 2 dimensional convolution, which inherits properties of 1 dimensional convolution that FIR filtering with 2 dimensional kernels in $x$ and $y$ direction use. Image (2 dimensional signal) is convolved (compute inner product) with FIR filter 2 dimensional signal (coefficients) sliding across the image. Since 1 dimensional convolution goes only into 1 direction, 2 dimensional as DFT goes in both (Equation~\ref{eq:conv2d}).

\begin{equation}
P_{x,y} * F_{x,y} = \sum_{n=0}^{N} \sum_{m=0}^{M} F_{m,n} \cdot P_{x-m, y-n}
\label{eq:conv2d}
\end{equation}

$P_{x,y}$ is image signal, $F_{x,y}$ is FIR filter window. Same as 1 dimensional convolution theorem that says that convolution in time domain corresponds to multiplication in frequency domain, same applies to image filtering. Labs 4 and 5 do not use frequency spectrum based filtering, thus I will not go into further detail.  Basic idea behind frequency domain filtering is instead of guessing values for perfect window FIR filter mask in spatial domain, simple rule for mask can be specified for frequency domain pairwise multiplication, like distance from origin to $(u, v)$ pair. Lowpass sets frequencies to 0 if distance is large, corresponding to attenuation of high frequency components.

Basic principle of filtering with spatial domain image kernels shown on mean window (Figure~\ref{fig:403}). With 3 by 3 kernel boundary pixels of image require padding of 1 which is done using different methods, depending on what is desired, such as padding with constant value replication padding, reflection mirror padding, or zero padding \footnote{\url{https://www.mathworks.com/help/images/imfilter-boundary-padding-options.html}}. Central pixel in 3 by 3 neighborhood gets average value. Convolution window continues sliding until all pixels in image receive average value. 

\begin{figure}[H]
  \includegraphics[width=0.6\textwidth]{image_105.png}
  \caption{mean kernel}\label{fig:403}
\end{figure}

Another kernels that I use in the lab 4 are median, laplacian, gaussian, and combined laplacian of gaussian.

Gaussian kernel is better version of mean window, because it gives more weight towards center pixel and direct neighborhood pixels. For example, in 3 by 3 kernel corner pixels receive weight = 1. This also helps to preserve object shape and edges better. Weight is controlled by $\sigma$ variable (Figure~\ref{fig:405}), such that smaller $\sigma$ results in more weight towards central pixel of convolution mask, and less towards neighborhood.

Median kernel replaces current center pixel with median value across its neighborhood. It is better than mean for noise reduction, because it removes small spikes caused by some reflections like stars on sky without blurring object edges so much as average does. 

Laplacian uses kernel values to determine image gradient in pixel intensity values in $x$ and $y$ direction (Figure~\ref{fig:404}). Instead of gradient vector $\nabla f(x,y) = g_x(x,y), \; g_y(x,y)$ that stores change in $x$ and $y$ directions separately, laplacian produces scalar by calculating horizontal and vertical second derivatives and adding together (Equation~\ref{eq:laplacian}). 

\begin{equation}
\nabla^2 f(x,y) = \frac{\partial^2 f(x,y)}{\partial x^2} + \frac{\partial^2 f(x,y)}{\partial y^2}
\label{eq:laplacian}
\end{equation}

If there is a change, edge pixels receive amplified brightness values. Along with thresholding applied it can draw object edges while making everything else completely black. 

\begin{figure}[H]
  \includegraphics[width=0.48\textwidth]{image_111.png}
  \caption{second derivative operator}\label{fig:404}
\end{figure}

Laplacian values are selected to represent second derivative, making it vulnerable to detect noise spike as real object edge. Therefore, gaussian is used together with laplacian to remove such noise from image, also called mexican hat operator (Figure~\ref{fig:405}).

\begin{figure}[H]
  \includegraphics[width=0.48\textwidth]{image_112.png}
  \caption{second derivative operator}\label{fig:405}
\end{figure}

There are simpler methods to manipulate images are based on pixel intensity values distribution. Histogram tells how much pixels from 0 (dark) to 255 (bright) are contained in image, equivalently histogram provides high level summary of image based on frequency of each brightness value in image. If image histogram turns out to be bipolar with 2 dominant peaks (a lot of bright and dark pixels), then simple tresholding could be used to assign 0 = Black for 1 group and 1 = White to another (Figure~\ref{fig:406}).

\begin{figure}[H]
  \includegraphics[width=0.6\textwidth]{image_104.png}
  \caption{image histogram}\label{fig:406}
\end{figure}

Another method of histogram modification is contrast enhancement that can be achieved with transfer functions and histogram equalization. 

Contrast is how well spread intensity values are in image, which is connected to how human eye perceives surroundings. If image is bright everywhere, it is difficult to see tiny details, as well as in complete dark image. In low contrast image histogram brightness values are clustered in a narrow range like most between 200-210 intensity levels, while other before and after this cluster are not used well. Thus, transfer function can be designed to spread out this narrow range to 0-255.

More formally, transfer function $T$ maps input intensity $r$ to output $s = T(r)$. Slope thus is multiplier of differences between pixel values. Where slope $>1$, input intensity range is stretched to wider output range and contrast increases. Where slope $<1$, range is compressed and contrast decreases (Figure~\ref{fig:407}). 

Gamma correction function uses exactly this principle. It is non linear transfer function to improve contrast in specific intensity range while compressing others. Output value is $s=g(x,y)$, input value is $r=f(x,y)$ which is original intensity value of pixel. Transfer function is $T[f(x,y)] = (f(x,y))^p$ applied to normalized pixel values in $[0, 1]$ range, where exponent $p$ determines shape of curve (Figure~\ref{fig:407}). If, for example, $p = 1$, $T$ is identity function with slope 1 everywhere, no change occurs. Only change in Matlab is naming convention $\gamma$ instead of $p$.

\begin{figure}[H]
  \includegraphics[width=0.6\textwidth]{image_106.png}
  \caption{transfer function and gamma correction}\label{fig:407}
\end{figure}

In probability density function $p_r(r)$ (chance of each brightness value occurance) total area under curve is always 1 because it represents all pixel intensity values. If section is stretched by slope $> 1$, width increases so height must decrease to keep area same. If compressed by slope $<1$, width decreases and height increases (Figure~\ref{fig:408}).

Histogram equalization achieves best possible contrast in image by using image intensity values histogram $p_r(r)$ to calculate custom transfer function that makes output histogram flat in density $p_s(s)$, equivalently uniform distribution with equal probability of each intensity level occurring. Cumulative distribution function needs to be applied as intensity transfer function. In continuous case with infinitely many brightness levels this is integration of $p_r(r)$. 

\begin{figure}[H]
  \includegraphics[width=0.46\textwidth]{image_108.png}\hspace{4pt}
  \includegraphics[width=0.50\textwidth]{image_109.png}
  \caption{continuous histogram equalization}\label{fig:408}
\end{figure}

For discrete 8-bit image with $L = 256$ gray levels, integration is replaced by discrete summation. Relative frequency of each intensity level $k$ is $p_r(r_k) = \frac{n_k}{n}$, where $n$ is total number of pixels and $n_k$ is number of occurrences of specific gray level $k$. Running sum of these probabilities gives each intensity level $k$ new rank (Equation~\ref{eq:histeq}).

\begin{equation}
s_k = T(r_k) = \sum_{i=0}^{k} p_r(r_i) = \sum_{i=0}^{k} \frac{n_i}{n}
\label{eq:histeq}
\end{equation}

Simple example of discrete histogram equalization for 8-bit image with 0-255 grey levels (Figure~\ref{fig:409}). Exactly such algorithm I used in lab. 

\begin{figure}[H]
  \includegraphics[width=0.6\textwidth]{image_110.png}
  \caption{discrete histogram equalization example}\label{fig:409}
\end{figure}

\subsubsection{(b) build the cumulative sum of this histogram}

I build cumulative sum vector by summing all histogram counts from 0 to $k$ at each position $k$. First element stays same as histogram, then each next adds previous cumulative value and current histogram count. Last value $cumulative(256)$ trivially equals total pixel count in image 69480.

\codeused
\begin{matlab}
cumulative = zeros(1, 256);
% pixel value 0 same
cumulative(1) = histogram(1);
% for each pixel value level between 1 and 255
for k = 2:256
    % old cumulative value + count of h
    cumulative(k) = cumulative(k - 1) + histogram(k);
end
\end{matlab}

\subsubsection{(c) normalise this cumulative sum to the max. pixel value}

Normalize calculated vector by dividing cumulative sum by its last value $cumulative(256)$ which equals total number of pixels. Result is 256 element vector with values from 0 to 1, which is exactly the cumulative distribution function $s_k = T(r_k)$ from Equation~\ref{eq:histeq}. Output on Figure~\ref{fig:411} shows that normalized cumulative sum is a very steep transfer function (will be used as lookup table) that maps cluttered 70-150 intensities value range to 0-1 taking all that range on $y$ axis.

\codeused
\begin{matlab}
% normalized 1*256 vector -> 256 values
% last value guaranteed to be max value
normalized = cumulative / cumulative(256);
figure; plot(0:255, normalized); title('cumulative sum of histogram');
\end{matlab}

For each pixel in image, old pixel value is used as index into normalized cumulative sum, and result is multiplied by 255 to get new pixel value. For example, pixel with old value 107 looks up {normalized(108) = 0.31}, multiplied by 255 this gives new value 79.

\subsection{Laboratory 4 -- Built your own gamma-correction function by using the template gctemp.m:}

In template $1/\gamma$ is used as exponent instead of $\gamma$ or $p$ that I used in theoretical introduction for transfer function (Figure~\ref{fig:407}). With $\gamma = 0.5$ function uses $r^{1/0.5} = r^2$ which compresses dark region details into tighter range. So, smaller $\gamma$ leads to increase of contrast for brighter image regions. When transfer function maps values outside of $[0, 255]$ range, output is clipped to boundary value 0 or 255. Instead of histogram equalization template also performs intensity values range contained in image linear stretching with Matlab stretchlim function \footnote{\url{https://www.mathworks.com/help/images/ref/stretchlim.html}}, so that it occupies full range between 0 and 1. For example, if image does not contain any intensity values below 100, it is a wasted area that can be occupied for better contrast.

\templateused
\begin{matlab}
function gctemp(image_path)
    function img_adj = myadjust(img, gamma)
        img_adj = img;
        % gamma correction
        img_adj = img_adj .^ 1/gamma;
    end
    ...
    function update()
        % without contrast stretch
        img_adj = myadjust(img, gamma);
        % with contrast stretch using stretchlim - CLIPPING!
        img_ref = imadjust(img, stretchlim(img), [0 1], gamma);
        ...
    end
    ...
end
\end{matlab}

I implemented 2 similar function according to template. First is basic gamma correction applied to image received in [0, 255] intensity value range. Second is doing stretching of intensity values to full [0, 1] range before applying gamma correction. 

Only difficulty was to implement stretchlim functionality that template uses with reference to Mathworks \footnote{\url{https://www.mathworks.com/help/images/ref/stretchlim.html}}. 

It is difficult to include both my function output and template, because to use template I need to make a call, and cannot view result in livescript output near plots of custom function. But based on my visual comparison results are identical, and $1/\gamma$ exponent in transfer function performs exactly as expected.

\codeused
\begin{matlab}
function output = gc(image, gamma)
    % normalize to [0, 1] range
    normalized = double(image) / 255;
    % apply gamma -> in script gamma >1 curve below 45 degree line darkens
    % image, however in template 1/gamma is used, thus to match behavior
    % for comparison..
    corrected = normalized .^ (1 / gamma);
    % back to [0, 255] range
    output = uint8(corrected * 255);
end
% template uses stretchlim(img) before applying gamma correction, makes
% contrast better, spreads out intensity values
% By default, the limits specify the bottom 1% and the top 1% of all pixel values.
% https://www.mathworks.com/help/images/ref/stretchlim.html
function output = gc_stretch(image, gamma)
    % normalize to [0, 1] range
    normalized = double(image) / 255;
    % contrast stretch to full [0, 1] range
    sorted = sort(normalized(:));
    % bottom 1% of all pixel values
    low = sorted(round(0.01 * length(sorted)) + 1);
    % top 1% of all pixel values
    high = sorted(round(0.99 * length(sorted)));
    % clip and stretch to [0, 1]
    normalized = (normalized - low) / (high - low);
    % clip values outside [0,1]
    normalized = max(0, min(1, normalized));
    % apply gamma
    corrected = normalized .^ (1 / gamma);
    % back to [0, 255] range
    output = uint8(corrected * 255);
end
\end{matlab}

\subsection{Laboratory 4 -- Perform different techniques on lowandsp.gif}

Membranes image as already discussed seems to have 2 major problems. 

First is noise spikes, that can be removed by convolution with 2 dimensional image filter. Since the goal is to preserve edges content, and noise is rare high frequency spikes, median 3 by 3 window would be a best approach. Mean would blur out edges and since noise spikes are rare very bright dots, their effect can be averaged and reduced, but not completely removed. This is exactly noise spike distortion pattern, as shown as example for median filtering on Mathworks \footnote{\url{https://www.mathworks.com/help/images/ref/medfilt2.html}}. In my median filter function I handle boundary pixels by skipping them, loop starts at (1 + edge) and ends at (rows - edge) so border pixels stay 0, since imfilt = zeros(rows, cols) initializes output as all zeros, so bounary pixels are never touched.

Second, is that image looks very grey, suggesting that histogram equalization is required to spread out intensity values and improve contrast, so that later on edges can be meaningfully amplified using Laplace operator.

\subsubsection{(a) to be finally able to highlight only the membranes and structures of the cells}

\begin{figure}[H]
  \includegraphics[width=0.6\textwidth]{image_62.jpg}
  \caption{original, mean filter, and median filter}\label{fig:422}
\end{figure}

As can be seen on output of Figure~\ref{fig:422} median filtering with 3 by 3 kernel perfectly removed all noise spikes completely without blurring membranes edges. This is exactly what expected, because if in 3 by 3 neighborhood 8 pixels have intensity value of 10 and only 1 spike has 255, median is 10, so 255 is completely removed and replaced with 10. Average kernel instead failed to remove such bright spikes only slightly averaging them out.  

\codeused
\begin{matlab}
cells = imread('lowandsp.gif');
% mean kernel for filtering
function imfilt = mean_filter(im, n)
    h = (1 / n) * ones(n, 1);
    H = h * h';
    % conv2 needs double
    % uint8 would clip fractional values during averaging
    imfilt = conv2(double(im), H, 'same');
    % convert back to uint8 to display
    imfilt = uint8(imfilt);
end
% median kernel for filtering 
function median = get_median(kernel)
    % sort values for median
    sorted = sort(kernel(:));
    % for 3 by 3 kernel ceil(9/2) = 5th vakue middle 
    % [1 2 3 4 -> 5th 6 7 8 9]
    n = length(sorted);
    median = sorted(ceil(n / 2));
end
function imfilt = median_filter(im, n)
    [rows, cols] = size(im);
    % INITIALIZE WITH ZEROS!
    imfilt = zeros(rows, cols);
    edge = floor(n / 2);
    % SKIP BOUNADRY PIXELS!
    for x = 1+edge:rows-edge
        for y = 1+edge:cols-edge
            kernel = im(x-edge:x+edge, y-edge:y+edge);
            imfilt(x, y) = get_median(double(kernel));
        end
    end
    imfilt = uint8(imfilt);
end
\end{matlab}

% just reuse equlization algortihm written from (a) to (d)
function image_equalization = histogram_equalization(im)
    image = im(:);
    histogram = zeros(1, 256);
    for idx = 1:length(image)
        histogram(image(idx) + 1) = histogram(image(idx) + 1) + 1;
    end
    cumulative = zeros(1, 256);
    cumulative(1) = histogram(1);
    for k = 2:256
        cumulative(k) = cumulative(k - 1) + histogram(k);
    end
    normalized = cumulative / cumulative(256);
    output = image;
    for k = 1:length(image)
        old_pixel = image(k);
        output(k) = normalized(old_pixel + 1) * 255;
    end
    image_equalization = uint8(reshape(output, size(im)));
end

For the edge detection I used laplacian and laplacian of gaussian discussed in theoretical introduction to highlight edges better. However a lot of tiny details on membrane, which are high frequency changes, distracted Laplacian and result is overamplified details across all image, not edges as I wanted (Figure~\ref{fig:424}). \codeused
\begin{matlab}
% laplacian for edge detection -> too sharpy
function imfilt = laplacian_filter(im)
    H = [0 1 0; 1 -4 1; 0 1 0];
    [rows, cols] = size(im);
    imfilt = zeros(rows, cols);
    for x = 2:rows-1
        for y = 2:cols-1
            imfilt(x, y) = sum(sum(double(im(x-1:x+1, y-1:y+1)) .* H));
        end
    end
    imfilt = abs(imfilt);
    imfilt = imfilt / max(imfilt(:)) * 255;
    imfilt = uint8(imfilt);
end
% laplacian of gaussian, so only important edges are highlited
function imfilt = log_filter(im, n, sigma)
    % gaussian filter
    G = zeros(n, n);
    center = floor(n / 2) + 1;
    for x = 1:n
        for y = 1:n
            G(x, y) = exp(-((x - center)^2 + (y - center)^2) / (2 * sigma^2));
        end
    end
    % K can be ignored to achieve amplification factor of 1 for DC, 
    % sum of coefficients must be equal to 1 -> normalize (eq 5.15)
    G = G / sum(G(:));
    smoothed = conv2(double(im), G, 'same');
    % laplacian filter -> pure high pass to highlight details and edges
    L = [0 1 0; 1 -4 1; 0 1 0];
    % conv2 does zero padding with 'same' parameter
    imfilt = conv2(smoothed, L, 'same');
    imfilt = abs(imfilt);
    imfilt = imfilt / max(imfilt(:)) * 255;
    imfilt = uint8(imfilt);
end

\subsection{Homework 4 (Diary only: Adapt your own histogram equalisation algorithm to perform a good result for the coloured picture church2.png)}

I used HSV color model (NB: worked better with this image than rgb2ycbcr with green channel for details) with rgb2hsv Matlab function \footnote{\url{https://www.mathworks.com/help/matlab/ref/rgb2hsv.html}}, since image is provided colored with 3 channels. Code is well commented, main difference from the lab is that I extract V = brightness channel and perform histogram equalization on it, which makes sense because these are color intensities. Only thing to track is converting between [0, 1] and [0, 255] as HSV returned as an m-by-n-by-3 numeric array with values in the range [0, 1] while developed equalization function are designed to receive input as [0, 255] and normalize after. Code is well commented, and other functionality is same as I used in lab.

\begin{figure}[H]
  \includegraphics[width=0.6\textwidth]{image_67.jpg}
  \caption{HSV color model based histogram equalization on church2.png}\label{fig:428}
\end{figure}

Output on Figure~\ref{fig:428} shows extremely good result with even dark grass getting enough contrast. My assumption is that given image is just artifically darkened with pixel intensities range compressed to a narrow. Thus, once equalization spreads them out, result is perfect.

\codeused
\begin{matlab}
% HSV -> color is defined by Hue, Saturation, Value (s 5.1, p. 233)
% the black and white image can be derived immediately from a color image,
% as the Value component corresponds to the brightness value
% -> equalize only V brightness channel, keep H and S unchanged
church = imread('church2.png');
% convert to HSV where V is brightness
churchhsv = rgb2hsv(church);
% https://www.mathworks.com/help/matlab/ref/rgb2hsv.html
% extract luminance channel
% HSV image, returned as 3-by-n-by-3 numeric array with values in range [0, 1]
% 1 hue, 2 saturation, 3 brightness
V = churchhsv(:, :, 3);
% convert from [0, 1] to [0, 255]
V = V * 255;
% convert to int
V = uint8(V);
% histogram of V channel
image = V(:);
histogram = zeros(1, 256);
for idx = 1:length(image)
    histogram(image(idx) + 1) = histogram(image(idx) + 1) + 1;
end
% cumulative sum
cumulative = zeros(1, 256);
cumulative(1) = histogram(1);
for k = 2:256
    cumulative(k) = cumulative(k - 1) + histogram(k);
end
% normalise to max pixel value
normalized = cumulative / cumulative(256);
% mapping from old to new pixel values
output = image;
for k = 1:length(image)
    old_pixel = image(k);
    output(k) = normalized(old_pixel + 1) * 255;
end
% put equalized V back as [0,1]
% keep H and S unchanged
church_hsv(:, :, 3) = double(reshape(output, size(V))) / 255;
church_eq = hsv2rgb(church_hsv);
figure; subplot(1, 2, 1); imshow(church); title('original');
subplot(1, 2, 2); imshow(church_eq); title('equalized');
\end{matlab}
